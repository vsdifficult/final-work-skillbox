# ADR 

- ADR-001 — Выбор архитектурного стиля: модульный монолит
Причина: меньшая сложность, быстрое MVP


Альтернативы: микросервисы с нуля


Почему отказано: слишком сложно и дорого на старте


Последствия: требуется план миграции → ADR-004


- ADR-002 — Выбор языка разработки: C#
Причины: экосистема .NET, простота модульного подхода, высокая производительность


Альтернативы: Node.js, Java


Последствия: требуются DevOps инструменты под .NET, контейнеризация



- ADR-003 — Выбор PostgreSQL для хранилища данных
Причины: надежность, расширяемость, универсальность


Альтернативы: MySQL, MongoDB


Последствия: необходимость проектирования схем заранее



- ADR-004 — План миграции от модульного монолита к микросервисам
Причины: постепенная масштабируемость


Последствия: необходимость API Gateway, сервисной коммуникации, отдельного логирования



- ADR-005 — Выбор REST как основного API
Причины: простота, стандартизированность


Альтернативы: gRPC


Последствия: больше сетевого трафика, но выше совместимость



- ADR-006 — Внедрение OAuth 2.0 + JWT для аутентификации
Причины: унифицированный способ безопасной авторизации


Последствия: нужна система ревокации и ротации токенов



- ADR-007 — Выбор Kafka/Webhooks для интеграций с внешними устройствами
Причины: высокая надёжность, асинхронность


Последствия: усложнение DevOps и мониторинга

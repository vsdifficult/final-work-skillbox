# Риски 

# Бизнес 
- Пользователи могут не возвращаться в приложение, если функционал недостаточно полезен или нет удерживающих механизмов.
- Рынок насыщен крупными игроками (Strava, Nike Training, Apple Health), что повышает стоимость привлечения клиентов 
- Некоторые функции зависят от производителей браслетов и трекеров. Изменение их API может замедлить развитие продукта. 
- Приложение может собирать много пользователей, но давать низкий доход без продуманной системы подписок и paid-функций.
- При работе с биометрическими и медицинскими данными требуется соблюдение законодательства (GDPR/ФЗ-152/другие нормы), нарушение — штрафы. 
- При расширении сервиса могут быть нужны дополнительные ресурсы: серверы, специалисты, маркетинг.

# Технические 
- Если изначально выбрана модульная/монолитная архитектура, нагрузка может вырасти быстрее, чем ожидается, что приведет к задержкам и падениям.
- Большое количество одновременных тренировок, трекинга датчиков и чатов может вызвать высокую нагрузку. 
- Открытые интеграции, обмен данными о здоровье и активностью пользователей создают риск утечек, атак и подмены данных. 
- Производители могут менять API, устройство может отправлять некорректные данные — это ломает работу трекинга. 
- Если данные с браслета/телефона приходят неравномерно, существует риск разрыва временной линии измерений.
- Без системы мониторинга, логов и метрик трудно быстро находить и устранять ошибки.
- Данные пользователя требуют шифрования, ротации ключей, специальных политик доступа. 
- Коммуникационный сервис может работать нестабильно под нагрузкой → падает качество пользовательского опыта.
